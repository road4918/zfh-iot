## 2. 核心模块详细需求

### 2.1 前置通信层（大规模接入重点）

| 功能项 | 需求描述 | 性能指标 |
|--------|----------|----------|
| **TCP Server** | 支持多端口监听，长连接保持，心跳检测 | 单机 10万+ 并发连接 |
| **UDP Server** | 支持无连接数据包接收，适用于上报型设备 | 单机 5万+ QPS |
| **MQTT Broker** | 可选集成，支持 Pub/Sub 模式 | 按 broker 规格 |
| **连接管理** | 设备认证（Device ID + Token）、IP 白名单、并发限流 | - |
| **负载均衡** | 支持多前置机横向扩展，设备路由策略 | 水平扩展 |
| **协议适配** | 支持多协议同时监听（不同端口/标识区分） | - |

### 2.2 前置业务层

| 功能项 | 需求描述 |
|--------|----------|
| **协议解析引擎** | 插件化架构，支持动态加载协议解析器 |
| **报文解码** | 将二进制报文解析为结构化数据 |
| **数据校验** | CRC 校验、长度校验、范围校验 |
| **指令下发** | 支持主动下发控制指令到指定设备 |
| **设备心跳** | 维护设备在线状态，超时离线判定 |

### 2.3 前置存盘层

| 功能项 | 需求描述 | 技术选型建议 |
|--------|----------|--------------|
| **数据缓冲** | 内存队列 + 磁盘持久化，防丢数据 | Disruptor / Kafka |
| **批量入库** | 批量写入，降低数据库压力 | 1000条/批次 |
| **时序数据库** | 存储抄表历史数据，支持时间范围查询 | InfluxDB / TDengine |
| **关系数据库** | 存储设备档案、配置信息 | MySQL / PostgreSQL |
| **数据归档** | 历史数据自动归档/压缩/清理 | 按时间策略 |

## 3. Web 管理后台

### 3.1 首页（数据大屏）

| 模块 | 展示内容 |
|------|----------|
| **网关统计** | 在线网关数 / 离线网关数 / 总数 |
| **设备统计** | 电表数 / 水表数 / 气表数 / 热表数 / 总数 |
| **数据完整率** | 今日应抄数 / 实抄数 / 完整率百分比 |
| **实时数据** | 最近上报的设备数据列表（滚动） |

### 3.2 系统管理

| 模块 | 功能点 |
|------|--------|
| **租户管理** | 多租户创建、租户配额管理、租户隔离 |
| **用户管理** | 用户增删改查、密码重置、状态启用/禁用 |
| **角色权限** | RBAC 模型、角色定义、菜单/按钮级权限控制 |
| **操作日志** | 登录日志、操作日志、异常日志、日志查询/导出 |
| **系统配置** | 平台参数配置、告警阈值配置、存储策略配置（详见附录 A） |

### 3.3 档案管理

| 模块 | 功能点 |
|------|--------|
| **网关管理** | 网关注册、网关信息维护、网关绑定/解绑、网关状态监控 |
| **表计管理** | 表计注册、表计参数配置（地址/协议/倍率）、表计绑定到网关 |
| **群组管理** | 设备分组、批量操作、分组统计 |
| **协议管理** | 协议配置、协议字段映射、协议版本管理 |
| **厂商管理** | 设备厂商档案、厂商协议关联 |

### 3.4 抄表数据

> **数据流向策略**：第一期先实现**设备主动上报**（设备→网关→平台），平台被动接收数据；第二期再实现**平台主动抄表**（平台下发指令→设备响应）。

| 模块 | 功能点 | 实现阶段 |
|------|--------|----------|
| **当前数据** | 实时数据查询、单设备详情、数据刷新 | Phase 4 |
| **历史数据** | 时间范围查询、曲线图展示、数据导出 Excel | Phase 4 |
| **抄表任务** | ~~定时抄表配置、补抄操作~~、抄表成功率统计 | Phase 5（平台抄表） |

> 注：数据修正功能暂不实现，以原始上报数据为准。

### 3.5 报警管理（暂不实现）

| 模块 | 功能点 | 状态 |
|------|--------|------|

| **报警记录** | 报警列表、报警确认、报警处理 | 暂不实现 |

## 4. 技术栈建议

| 层级 | 技术选型 |
|------|----------|
| **前置通信** | Netty（TCP/UDP）、EMQX（MQTT可选） |
| **协议解析** | Java SPI 插件机制，支持动态加载 |
| **缓存/队列** | Redis、Caffeine、Disruptor |
| **时序数据库** | TDengine（国产，适合物联网场景） |
| **关系数据库** | MySQL / PostgreSQL |
| **后端框架** | Spring Boot + MyBatis-Plus |
| **前端框架** | Vue3 + Element Plus |
| **权限框架** | JWT + Shiro |

## 附录 A：术语说明

### A.1 操作日志中的异常日志

**异常日志** 是指记录系统中发生的非正常情况的日志，具体包括：

| 异常类型 | 说明 | 示例 |
|---------|------|------|
| **系统异常** | 程序运行时抛出的未捕获错误 | 数据库连接失败、空指针异常、数组越界等 |
| **业务异常** | 违反业务规则的操作 | 设备不在线却尝试下发指令、重复注册已存在的设备等 |
| **安全异常** | 潜在的安全威胁行为 | 非法登录尝试、越权访问、SQL 注入尝试等 |
| **通信异常** | 设备通信过程中的异常 | 设备连接超时、协议解析失败、报文校验错误等 |
| **数据异常** | 数据处理过程中的问题 | 数据格式不正确、数值超出合理范围、数据丢失等 |

**与操作日志的区别：**
- **操作日志**：记录用户的正常操作行为（如"用户 admin 查询了设备列表"）
- **异常日志**：记录非正常情况，需要运维人员关注并处理的错误事件

### A.2 系统配置中的存储策略配置

**存储策略配置** 是指对物联网平台数据存储生命周期和存储方式的管理配置，主要包括：

| 配置项 | 说明 | 典型配置 |
|--------|------|---------|
| **数据保留时长** | 不同类型数据的存储周期 | 原始数据保留 30 天，小时级数据保留 90 天，日级数据保留 3 年 |
| **存储分层** | 数据在不同存储介质的分布 | 热数据存 SSD（近 7 天），温数据存 HDD（7-30 天），冷数据存对象存储（30 天以上） |
| **自动归档** | 历史数据的自动迁移和压缩 | 超过 30 天的数据自动归档到低成本存储，超过 1 年的数据压缩存储 |
| **清理策略** | 过期数据的自动清理规则 | 原始抄表数据保留 1 年后删除，报警记录保留 2 年后删除 |
| **备份策略** | 数据备份的频率和保留策略 | 每日全量备份，保留最近 30 天的备份，每月保留一份存档备份 |

**为什么需要存储策略配置？**
- 物联网数据量巨大，合理的存储策略可以**降低 60-80% 的存储成本**
- 保证**热数据的快速查询**性能
- 满足**数据合规**要求（如某些行业要求数据必须保留 X 年）
- 防止存储无限增长导致**系统性能下降**
